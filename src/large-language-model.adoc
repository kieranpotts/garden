= Large language model (LLM)

A large language model (LLM) is a neural network trained on very large text datasets to predict and generate language. Modern LLMs can perform many tasks, including answering questions, document summarization, translation, and code generation.

toc::[]

== How LLMs work

=== Training

In the development of LLMs, training is the computationally-enormous process of adjusting a model's weights by exposing the model to vast amounts of data, calculating how wrong the model's outputs are, and nudging the parameters in the right direction… billions of times over.

Training a frontier model like Llama or Qwen requires data centres full of specialized hardware running for weeks or months.

Training demands maximum compute throughput and benefits enormously from features like high-bandwidth inter-GPU connections.

=== Fine-tuning

Fine-tuning is the process of taking a pre-trained model and continuing to train it on a smaller, task-specific dataset to adapt its behaviour for a particular domain or use case. Rather than training from scratch — which requires enormous compute and data — fine-tuning starts from an existing model's weights and adjusts them incrementally.

Fine-tuning is how general-purpose base models are turned into specialised assistants. A model fine-tuned on medical literature will perform better on clinical tasks than the same base model used out of the box.

A lighter-weight alternative is *prompt-based adaptation*. This is the process of crafting system prompts and few-shot examples to steer model behaviour without changing its weights at all. This is simpler but less powerful than fine-tuning. Ollama's Modelfiles are an example of prompt-based adaptation.

=== Quantisation

Quantisation is a technique for reducing the numerical precision of a model's weights — for example, representing values with 4-bit integers instead of 16-bit or 32-bit floats. This reduces memory footprint and speeds up inference at the cost of a small reduction in output quality.

Quantisation is what makes it practical to run large models on consumer hardware with limited VRAM.

=== Inference

In the context of LLMs, inference simply means using a trained model to generate output. It is distinct from training, which is the process of creating the model in the first place.

The term comes from logic and statistics, where "inference" means drawing a conclusion from available information. In LLMs, inference is the process of calculating the next token, based on past learning.

When you type a prompt into an LLM and get a response back, that entire process – loading the model, processing your input, generating each token of the reply – is inference. The model's weights – the billions of numerical parameters that encode everything it learned during training – are fixed. Inference is the process of running data through those fixed weights to produce an output.

Compared to training, inference is relatively cheap. It can be done on a single workstation GPU, with available VRAM being the primary constraint on performance and the size of the model you can use.

=== Tokens

LLMs do not process text character by character or word by word — they work in tokens.

A token is a chunk of text, typically a word, part of a word, or a punctuation character. Common words are usually a single token; rarer or longer words may be split into multiple tokens. As a rough guide, one token is approximately four characters or three-quarters of a word in English.

Tokens are the unit of measurement for both input (the prompt) and output (the response). LLM pricing is typically quoted per million tokens, and model capabilities such as context window size are measured in tokens.

=== Context window

The context window is the maximum number of tokens an LLM can process in a single interaction — the combined total of the system prompt, conversation history, and the current input and output. Text outside the context window is not visible to the model.

Context window size is one of the most practically important model characteristics. A larger context window allows the model to consider more of a document, a longer conversation history, or a larger codebase at once.

Modern frontier models support context windows of 128K tokens or more. Some extend to 1M tokens or beyond.

=== Retrieval-augmented generation (RAG)

A core limitation of LLMs is that their knowledge is frozen at the point of training. By default, models cannot access new information or browse the web. Retrieval-augmented generation (RAG) is a technique that addresses this by combining an LLM with an external knowledge source, retrieved at inference time.

In a RAG system, a user's query is used to search a knowledge base (a vector database, a search index, or live web results). The most relevant results are fetched and injected into the model's context alongside the original query. The model then generates a response grounded in that retrieved content, rather than relying solely on what it learned during training.

RAG is widely used to build knowledge-aware applications — internal document Q&A systems, customer support bots, and AI search engines. https://www.perplexity.ai/[Perplexity] is a prominent consumer example. It uses RAG over live web search to answer questions with cited sources in real time. See *link:./ai-assistant.adoc[AI assistants]* for other examples.

=== Compute

Both training and inference are computationally intensive and rely heavily on GPUs, which excel at the massively parallel matrix operations that underpin neural networks.

The dominant frameworks for programming GPUs are:

* https://developer.nvidia.com/cuda-toolkit[CUDA] — NVIDIA's proprietary parallel computing platform, dominant in AI/ML workloads due to its maturity and deep integration with frameworks like PyTorch and TensorFlow. Requires NVIDIA hardware.

* https://www.amd.com/en/products/software/rocm.html[ROCm] — AMD's open-source GPU computing platform, their equivalent of CUDA for AMD hardware. Growing PyTorch and TensorFlow support, but still lags behind CUDA in ecosystem maturity.

* https://www.khronos.org/opencl/[OpenCL] — An open standard for parallel computing across heterogeneous hardware (CPUs, GPUs, FPGAs) from any vendor. More portable than CUDA but less widely adopted in the AI ecosystem.

=== Inference engines

Inference engines are the low-level runtimes that actually execute a model — loading weights into memory and running the computations that produce each token. Model managers like Ollama and LM Studio (see above) are built on top of them.

The two most widely used inference engines for open-weight models are:

* https://github.com/ggml-org/llama.cpp[llama.cpp] — A C/C++ inference engine optimised for running quantised models on consumer hardware, including CPU-only machines, Apple Silicon, and devices with limited VRAM. Uses the GGUF model format. It is the engine that powers Ollama and LM Studio. Prioritises broad hardware compatibility over maximum throughput.

* https://github.com/vllm-project/vllm[vLLM] — A Python-based inference engine designed for high-throughput production serving on GPU servers. Its key innovation is _PagedAttention_, which manages the KV cache more efficiently, enabling higher request concurrency. Supports multi-GPU tensor parallelism and exposes an OpenAI-compatible API. This is a good choice when serving many users concurrently, rather than running models locally for a single user.

== Choosing LLMs

In choosing an LLM, you need to appreciate the tradeoffs to be made between capability, size, speed, and cost. The bigger the AI model, the more capable it is. But the trade-off is that the big, advanced "frontier" models are also slow and costly to run.

At the other end of the spectrum are small models that are optimized for speed rather than capability. Some of these small models may also specialize in particular domains, rather than being capable of lots of different tasks.

Somewhere in the middle are the general-purpose models that find a good balance between these factors, and can do most tasks well.

=== Frontier models

A frontier model is a model that represents the current state of the art — the most capable models available at a given point in time.

Frontier models are typically the largest models, trained on the most data, with the best performance across a broad range of tasks. They are developed by well-resourced AI labs and accessed via paid APIs rather than downloaded and run locally.

Frontier models are usually proprietary — which means their weights are not publicly released. The companies that develop them invest heavily in safety evaluation and alignment — the process of steering model behaviour to be helpful, honest, and to avoid harmful outputs.

Modern frontier models are increasingly *multimodal* — capable of processing not just text but also images, audio, and video as inputs, and in some cases generating them as outputs.

Examples of frontier model families include:

* https://platform.openai.com/docs/models[OpenAI's GPT models].
* https://www.anthropic.com/claude[Anthropic's Claude models].
* https://deepmind.google/technologies/gemini/[Google's Gemini models].

=== Open-weight models

*Open-weight* models have publicly released weights, meaning anyone can download and run them. This is distinct from *open-source*, which would also require the training data and code to be public — most open-weight models do not meet that bar.

Licensing terms vary: some permit commercial use, others restrict it.

https://huggingface.co/[Hugging Face] is the primary hub for discovering and downloading open-weight models, hosting over two million models contributed by individuals, research labs, and major organisations.

As of 2026, some notable open-weight model families include:

* The https://llama.meta.com/[Llama] models from Meta.
* The https://qwenlm.github.io/[Qwen] models from Alibaba.
* The https://huggingface.co/moonshotai[Kimi] models from Moonshot AI.
* The https://huggingface.co/THUDM[GLM] models from Z.ai.
* The https://www.deepseek.com/[DeepSeek] models from DeepSeek AI.

== Model access layers

=== Model gateways

Model gateways provide a single, unified API endpoint, and/or a GUI or other user interface, to access models from multiple hosted LLM providers. Rather than integrating directly with each provider's SDK and managing separate authentication, response formats, and failover logic, developers integrate once with the gateway, which handles routing, fallback, cost optimisation, and observability across providers.

* https://openrouter.ai/[OpenRouter] — Unified API for 300+ models across multiple providers, with per-request routing, cost tracking, and usage rankings.

* https://anannas.ai/[Anannas] — OpenAI-compatible API gateway with intelligent routing, automatic failover, and real-time cost dashboards.

* https://huggingface.co/inference-api[Hugging Face Inference API] — Managed inference endpoints for models hosted on Hugging Face, with serverless and dedicated deployment options.

* https://www.perplexity.ai/[Perplexity] — AI-native search engine and general-purpose *link:./ai-assistant.adoc[AI assistant]. Provides access to a large number of models.

[TIP]
======
A single subscription to a gateway service like Perplexity Pro or OpenRouter can be more economical than maintaining separate subscriptions with multiple AI labs. These services give you access to models from OpenAI, Anthropic, Google, and others through one account, letting you switch between them based on the task at hand.
======

=== Model managers

Model managers are primarily designed for running open-weight models locally. They handle downloading of models, managing hardware resources, and serving a local inference endpoint — typically an OpenAI-compatible API on localhost.

Open-weight models, operated through local model managers, are a good choice for privacy-sensitive use cases, offline environments, or experimenting with open models without incurring API costs.

By contrast, *model API gateways* (above) are middleware for routing requests to remote, hosted models across multiple cloud providers.

Some popular model managers include:

* https://ollama.com/[Ollama]
* https://lmstudio.ai/[LM Studio]
